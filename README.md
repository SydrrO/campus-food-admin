# 校园餐饮商家后台

版本：`v2.3`

这是餐饮商家后台的升级源码仓库，当前重点维护后台管理页面、优惠券管理、订单筛选和用户数据展示。仓库只保存源码，不保存数据库、证书、SSH 私钥或服务器密码。

## v2.3 主要内容

- 后台主页面标题更新为 `v2.3`。
- 订单列表底部改为卡片式工单展示，联系人、电话、状态、餐别、配送、菜品、金额、备注和操作按钮分区呈现。
- 菜品明细改为标签展示，长订单号压缩显示，手机端按钮两列排布，减少底部订单挤压和混杂感。
- 本版按要求同步 GitHub 并创建 `v2.3` Release，包含 `v2.1` 到 `v2.3` 的最新后台改造。

## v2.2 主要内容

- 后台主页面标题更新为 `v2.2`。
- 优惠金额覆盖订单应付金额时，订单实付记为 `0.00`，直接确认为已支付/已确认，不再调用微信支付统一下单。
- 0 元优惠订单会正常写入订单、核销优惠券、保留系统记录，并跳过 Redis 支付超时队列。
- `/payment/prepay` 对已完成的 0 元优惠订单增加保护，误调时返回 `mode=free`，不会生成微信支付参数。
- 本次后端逻辑已上线，Release 合并到 `v2.3` 统一发布。

## v2.1 主要内容

- 后台主页面标题更新为 `v2.1`。
- 顶部“赛杜后台”放大，并改为暗金与蒂芙尼蓝交织渐变，版本号缩小跟随展示。
- 增加克制的交互动效：按钮、菜单、弹窗、页面切换和用户卡片有轻微过渡反馈，并兼容系统减少动态设置。
- 本次为 `v2.1` 小版本，按“五个小版本上传一次 GitHub”的规则暂不单独同步 GitHub。

## v2.0 主要内容

- 后台主页面标题更新为 `v2.0`。
- 配送详情弹窗去掉底部重复关闭按钮，只保留顶部关闭。
- 订单筛选区恢复为两列两排布局，按钮独立占下一排，手机端配送日期不再挤出屏幕。
- 顶部“赛杜后台”改为暗金色品牌字，版本号缩小跟随展示。
- 用户列表改为新的卡片式展示，头像、昵称、ID、消费和更多按钮分区清晰，不再沿用表格压缩布局。
- 从 `v1.6` 到 `v2.0` 已累计 5 个小版本，本版适合统一同步 GitHub 并创建 Release。

## v1.9 主要内容

- 后台主页面标题更新为 `v1.9`。
- 用户列表手机端重新排版：左头像、中间昵称与 ID/消费、右侧更多，避免控件混在一起。
- 用户行增加更清晰的独立边框、留白和左侧金色细线，整体更接近高奢名单卡。
- 头像无图片时只显示首字符，不再出现被省略号污染的头像文字。
- 用户 ID 和累计消费在同一行内等宽居中，金额不再和头像或按钮打架。
- 订单筛选区手机端改为单列等宽输入框，并针对 iOS Safari 日期控件取消原生撑宽。

## v1.8 主要内容

- 后台主页面标题更新为 `v1.8`。
- 用户数据页整理为高奢名单册风格：边框更明确，ID 和累计消费居中对齐。
- 用户长昵称显示时中间省略，保持列表字符长度和视觉节奏一致。
- 用户搜索、重置、0 消费展开/折叠移入“筛选/显示”弹窗，减少列表页杂乱感。
- 菜单数据页移除大块分类管理区，分类改为在菜品编辑弹窗中选择。

- 订单筛选栏的关键词、配送日期、餐别、状态输入框统一宽度，手机端日期框不再撑出屏幕。

## v1.7 主要内容

- 后台主页面标题更新为 `v1.7`。
- 全后台增加手机窄屏优化：顶部、统计卡、筛选区、按钮、订单卡片、菜单菜品卡、优惠券表格和弹窗在窄屏下更紧凑。
- 用户列表在手机端改为卡片式展示，减少横向滚动。
- 优惠券列表和已使用券弹窗在手机端改为带字段名的卡片式展示。
- 优惠券发放页支持按用户 ID 或昵称搜索用户；昵称匹配到多人时会提示输入更完整信息。
- 订单处理统计取消“待支付”，保留当前结果、已确认、处理中、已关闭。

## v1.6 主要内容

- 后台主页面标题更新为 `v1.6`。
- 后台默认首页改为订单界面。
- 原本四个大页面入口收进右上角三点菜单，菜单内包含订单、用户、菜单、优惠券。

- 主内容区改为无侧栏的全宽布局，手机端只保留右上角菜单入口，减少挤压。

## v1.5 主要内容

- 后台主页面标题和左侧名称更新为 `v1.5`。
- 积分余额不再被消费金额刷新逻辑覆盖，用户兑换优惠券或后台扣减后会保留扣减结果。
- 支付成功仍按实付金额乘以 `100` 增加积分。
- 订单退款时只撤回该订单曾发放的积分一次，避免重复扣减。

## v1.4 主要内容

- 后台主页面标题和左侧名称更新为 `v1.4`。
- 订单筛选区按钮改为一排：查询、重置、复制回执、打单。
- “复制回执”会一次复制午餐和晚餐商家回执；若没有晚餐有效订单，则不写晚餐段落。
- 手机端订单筛选按钮改为紧凑四列，减少换行和挤压。

## v1.3 主要内容

- 用户列表仍默认折叠 0 消费用户，但搜索时会自动展示全部匹配用户，避免查不到 0 消费用户。
- 用户积分按已支付订单实付金额乘以 `100` 发放，每次支付入账时同步增加。
- 用户消费与积分刷新逻辑统一校准，退款或手动退款后会按有效订单重新计算。
- GitHub 改为每累计 5 个小版本统一上传和创建 Release。

## 验收清单

1. 打开后台主页，顶部应显示 `赛杜后台 v2.3`，默认进入订单界面。
2. 进入用户数据页，用户卡片只展示头像、用户 ID、昵称、累计消费、操作。
3. 用户 ID 应为四位数格式，例如 `0001`。
4. 会员头像应有金色边框。
5. 点击用户行尾“更多”，弹窗应显示默认地址电话、积分、订单数、创建时间；未下过单的用户电话留白，会员用户额外显示会员开通时间。
6. 进入优惠券页面，应可通过用户 ID 或昵称查询用户；昵称匹配多人时应提示输入更完整信息。
7. 进入优惠券页面，已使用券默认不在列表中展示。
8. 点击“查看已使用券”，应弹窗展示该用户已使用优惠券。
9. 进入订单页，不手动选择筛选条件时，应按当前时间自动套用默认日期、午餐、已确认。
10. 进入订单页，点击“配送详情”，弹窗应按当前筛选显示每个配送区域分别几单。
11. 用户列表应默认折叠 0 消费用户，剩余用户按累计消费金额降序排列。
12. 搜索用户时，匹配到的 0 消费用户也应展示出来。
13. 新支付订单应按实付金额乘以 `100` 增加用户积分，退款时只撤回对应订单曾发放的积分。
14. 订单页筛选按钮应在一排显示；复制回执应包含午餐，且仅在有晚餐有效订单时追加晚餐。
15. 用户消费积分后，刷新消费金额或退款同步不应把积分余额覆盖回累计消费对应值。
16. 右上角三点菜单应能打开订单、用户、菜单、优惠券四个入口，手机端不应挤压页面标题。
17. 手机窄屏下用户列表和优惠券列表应卡片化展示，页面不应出现主要内容横向溢出。
18. 订单处理统计区不再显示“待支付”统计。
19. 用户数据页 ID 和累计消费应居中对齐，长昵称应中间省略且列表不杂乱。
20. 用户搜索和 0 消费显示控制应通过“筛选/显示”弹窗完成。
21. 菜单数据页不再显示大块分类管理区；新增栏目应从菜品编辑分类下拉进入。
22. 订单筛选栏日期框在手机端不应超出屏幕，四个筛选框宽度应保持一致。
23. 手机端用户列表每行应清晰分区：头像不压住 ID，ID 与消费居中齐平，右侧更多按钮独立。
24. 手机端订单筛选区应两列两排等宽显示，配送日期框不得突破面板边界。
25. 配送详情弹窗只保留一个关闭按钮。
26. 订单筛选区应显示为两列两排，操作按钮在筛选项下方单独一排。
27. 顶部品牌应显示暗金色“赛杜后台”，版本号以更小字号跟在旁边。
28. 用户列表应为独立卡片展示，头像、昵称、ID、消费和更多按钮不得互相挤压或重叠。
29. 顶部“赛杜后台”应比旧版更醒目，颜色为暗金与蒂芙尼蓝交织渐变，版本号保持小号跟随。
30. 按钮、菜单、弹窗和页面切换应有轻微动效反馈，但不得出现夸张跳动或花哨动画。
31. 当优惠金额大于或等于订单应付金额时，创建订单应返回 `status=confirmed`、`actual_amount=0.00`、`expire_time=null`，且不应调用微信支付统一下单。
32. 订单列表底部应按卡片式工单展示，手机端联系人、配送、菜品、金额和操作按钮不应挤在一起。

## 部署说明

当前这版更新订单列表卡片展示、后台版本号和主页面静态资源缓存号，部署时应先备份服务器已有文件，再覆盖以下文件：

- `admin/index.html`
- `admin/coupons.html`
- `admin/css/workbench.css`
- `admin/js/workbench.js`
- `admin/js/coupons.js`

这是纯前端静态文件更新，部署后通常不需要重启后端服务。

## 注意事项

- 不要把密钥、数据库、证书、服务器密码提交到仓库。
- 不要覆盖服务器上无关项目。
- 每次上线前先做服务器备份，出问题可以直接回滚静态文件。
